architecture · three views

How it fits together, visually

view 1

The bridge — an adapter

Harbor only knows its own environment interface. The bridge implements that interface and quietly forwards every call to a managed ZenML Sandbox session — so nothing in Harbor changes, but everything now runs (and is observed) through ZenML.

view 2

The Sandbox component — one interface, many backends

The Sandbox is a ZenML stack component, like an orchestrator or artifact store. It's an abstraction (BaseSandbox) with swappable flavors. Change the flavor and the same campaign runs on Modal, locally, or your K8s — no code change.

view 3

All together — three planes

Govern centrally, orchestrate the matrix, execute on your substrate. The pipeline spins sandboxes through the component; the job tree and rewards come back through the steps, and artifacts & lineage land in the control plane — one place to see and govern every eval.

under the hood

One interface, not a rewrite

ZenML implements Harbor's BaseEnvironment. Harbor thinks it's talking to a normal environment; every call is actually a managed ZenML Sandbox session — which is how observability and governance come for free.

The environment bridge

Harbor BaseEnvironment → ZenML SandboxSession
start(force_build)→create_session() · ensure /logs · replay Dockerfile
exec(cmd, cwd, env)→session.exec(["bash","-c",…]) · stdout/stderr → ZenML log source
upload_file / upload_dir→session.upload_file (dirs tar'd)
download_file / download_dir→session.download_file → versioned artifact
stop(delete)→session.destroy() / close()
Because Harbor pulls its results out via download, the whole job tree passes through ZenML on the way out — so it lands as a versioned artifact, not a file on a disappearing pod.

Honoring the task's image

task environment/Dockerfile → sandbox, at runtime
FROM python:3.12-slim→base_image — pulled by the flavor
WORKDIR /app→mkdir + default cwd for every exec
RUN …→exec() replayed right after start
COPY src dst→upload from the build context
The trial runs in the task's own image — the component pulls it and replays setup via the session. No ZenML dependencies are baked into eval images.

In code it's about ten lines — a generic fan-out you write once (top), and the one import_path that routes Harbor through the bridge (bottom):

① the campaign · harbor_eval_campaign.py　　② the trial · run_harbor_job.py
# Fan out one tracked, retryable step per (agent × model × task)
job_dirs  = run_harbor_job.map(spec=matrix)
summaries = parse_harbor_job.map(job_dir=job_dirs, spec=matrix)
return build_report(job_summaries=summaries)   # → CampaignReport + HTML

# Inside run_harbor_job — Harbor runs *through* the bridge:
config = JobConfig(
    tasks=[TaskConfig(path=task)],
    agents=[AgentConfig(name=agent, model_name=model)],
    environment=EnvironmentConfig(
        import_path="zenml_sandbox_env:ZenMLSandboxEnvironment"),  # ← the bridge
    verifier=VerifierConfig(),
)
await Job.create(config).run()   # trial executes in the task's own image
